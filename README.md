# Renga Script Manager

Менеджер скриптов для Renga, который позволяет создавать кнопки, меню и выпадающие списки без необходимости перекомпиляции плагина. Вы просто создаете папки и кладете туда C# скрипты (`.cs`).

## Структура папок

Плагин автоматически сканирует папку `Scripts` внутри своего расположения и строит интерфейс на основе найденных папок:

```text
📂 Scripts
 ┣ 📂 Моя_Кнопка.button              <-- Одиночная кнопка на главной панели
 ┃  ┣ 📄 script.cs
 ┃  ┣ 📄 UI.cs                       <-- (Новое!) Можно класть сколько угодно .cs файлов
 ┃  ┗ 📄 icon.png
 ┣ 📂 Выпадающий_Список.folder       <-- Выпадающий список на главной панели
 ┃  ┣ 📄 config.json
 ┃  ┣ 📂 Пункт1.button
 ┃  ┃  ┗ 📄 script.cs
 ┃  ┗ 📂 Пункт2.button
 ┃     ┗ 📄 script.cs
 ┣ 📂 Правый_Клик.context            <-- Меню по правому клику мыши
 ┃  ┣ 📄 config.json
 ┃  ┣ 📂 Пункт1.button
 ┃  ┃  ┗ 📄 script.cs
 ┃  ┗ 📂 Подменю.folder              <-- (Новое!) Многоуровневые контекстные меню
 ┃     ┣ 📄 config.json
 ┃     ┗ 📂 Вложенная_кнопка.button
 ┃        ┗ 📄 script.cs
 ┣ 📂 Панель_Действий.actions        <-- Боковая панель действий (при выделении объекта)
 ┃  ┣ 📄 config.json              
 ┃  ┣ 📂 Кнопка.button
 ┃  ┃  ┗ 📄 script.cs
 ┃  ┗ 📂 Список.folder               <-- Выпадающие списки внутри панели действий
 ┃     ┗ ...
 ┗ 📂 lib                            <-- Общие библиотеки (DLL и C# код)
    ┣ 📄 EPPlus.dll
    ┗ 📄 MyHelpers.cs
```

### Типы поддерживаемых элементов:
1. **`.button`** — Одиночная кнопка. Внутри должен быть хотя бы один файл `.cs`. Если их несколько, они склеятся автоматически!
2. **`.folder`** — Выпадающий список или подменю. У выпадающих списков на панелях текст скрыт, показывается только иконка и ToolTip.
3. **`.context`** — Корневое меню по правому клику.
4. **`.actions`** — Панель действий Renga.

## Настройки внешнего вида (`config.json`)

В каждой папке вы можете создать файл `config.json` (в кодировке **UTF-8 без BOM**). Этот файл управляет отображением элементов.

### Базовые параметры (для `.button`, `.folder`, `.actions`, `.context`)
```json
{
  "Title": "Моя Кнопка",
  "ToolTip": "Развернутое описание того, что делает эта кнопка"
}
```
* **`Title`** — Отображаемое имя (вместо имени папки).
* **`ToolTip`** — Всплывающая подсказка при наведении мыши (работает везде и это очень полезная функция!).

### Параметры `ViewType` и `ShowCase` (для `.actions` и `.context`)

Когда вы создаете панель действий (`.actions`) или контекстное меню (`.context`), вам нужно указать Renga, **где именно** их показывать.

```json
{
  "Title": "Мое Супер Меню",
  "ViewType": "View3D",
  "ShowCase": "Scene"
}
```

#### Доступные значения `ViewType` (в каком окне показывать):
* `Undefined` — показывать везде *(Работает только для `.context`! Для `.actions` этот параметр не работает, нужно указывать конкретное окно)*.
* `View3D` — только на 3D Виде.
* `Drawing` — только на Чертеже.
* `Level` — только на Уровне (в плане).
* `ProjectExplorer` — в Обозревателе проекта.
* `Section` — на Разрезе.
* `Facade` — на Фасаде.
* `Table` — в Таблице.
* `Specification` — в Спецификации.
* `SectionProfile` — в Профиле сечения.
* `Assembly` — в Сборке.
* `PipeSystem` — в Системе трубопроводов.

#### Доступные значения `ShowCase` (только для `.context` меню):
* `Selection` — меню появится, только если вы кликнули правой кнопкой мыши **по выделенному объекту** (по умолчанию).
* `Scene` — меню появится, если вы кликнули правой кнопкой мыши **в пустом месте сцены/вида** (не выделяя объект).

*(Для `.actions` параметр `ShowCase` игнорируется, так как панель действий по своей природе появляется только при выделении `Selection`)*

### Порядок кнопок и линии-разделители
Чтобы жестко задать порядок кнопок или добавить разделитель, добавьте массив `"Items"` в `config.json` **родительской** папки.
*(Кстати, вы можете создать `config.json` прямо в самой главной папке `Scripts`, чтобы отсортировать глобальные меню и панели на самом верхнем уровне!)*

```json
{
  "Items": [
    "Создать.button",
    "---",
    "Удалить.button",
    "Анализ.folder"
  ]
}
```
*Кнопки, которые есть на диске, но не указаны в `Items`, будут добавлены в конец списка по алфавиту.*
*Разделители `"---"` поддерживаются только в `.folder` и `.context`.*

## Иконки
Положите файл `icon.png` (рекомендуется **24x24 px**) внутрь папки `.button` (рядом со скриптом). 
Вы также можете положить `icon.png` внутрь папки `.folder`, чтобы задать иконку для выпадающего списка на панелях!

## Загрузка скриптов по сети (Remote Scripts)

Плагин `ScriptManager` поддерживает гибридный режим работы: он может загружать исполняемый код `C#` напрямую с удаленного сервера (как `script.py` в pyRevit). Это избавляет от необходимости распространять файлы с кодом локально.

### Как это настроить:

1. **Глобальный конфиг сервера:**
   Создайте файл `server_config.json` в самом корне папки `Scripts` **или** прямо рядом с файлом плагина `ScriptManagerPlugin.dll`:
   ```json
   {
     "BASE_URL": "http://10.2.176.2",
     "USERNAME": "optional_login",
     "PASSWORD": "optional_base64_password"
   }
   ```
   *Если возникнут проблемы со связью (сервер недоступен, неверный пароль, или файла нет на сервере), плагин не будет выводить страшные логи в консоль, а покажет аккуратное системное окошко с понятной ошибкой (например, "Нет прав доступа к серверу!").*
2. **Локальная папка-пустышка:**
   Создайте папку нужной кнопки, например `Scripts\Моя_Кнопка.button`. Вы можете положить туда настройки и иконку (`icon.png`, `config.json`, `remote_files.json`), но **ни одного локального файла `.cs`**. 
3. **Файлы на сервере:**
   На вашем веб-сервере создайте папку `SMPRenga`. Воспроизведите там структуру папок и положите файл `script.cs`.
   *Пример:* Для локальной папки `Scripts\Моя_Кнопка.button` плагин будет скачивать код по адресу: `http://10.2.176.2/SMPRenga/Моя_Кнопка.button/script.cs`.

### Загрузка нескольких файлов (Манифест)
Если вашему удаленному скрипту требуется несколько файлов (а не только `script.cs`), вы можете создать в локальной папке-пустышке файл `remote_files.json`.
```json
[
  "main.cs",
  "ui.cs",
  "helpers.cs"
]
```
При нажатии на кнопку плагин скачает **все** указанные файлы с сервера и разом скомпилирует их в единую сборку!

*Примечание: Плагин обладает умным кэшированием. Он скачивает код с сервера при каждом клике, но компилирует его заново (что занимает ~1 секунду) только в том случае, если исходный код на сервере действительно изменился. Если код тот же самый — кнопка срабатывает мгновенно из оперативной памяти.*

## Как писать скрипты (`script.cs`)

Код пишется на чистом C#. Менеджер сам компилирует его "на лету". Поддерживается **два стиля написания скриптов** (плагин автоматически распознает стиль).

### Доступные команды:
1. `RengaApp` — готовая ссылка на `Renga.IApplication`.
2. `Print("текст")` — вывод текста в консоль скриптов.
3. `SafeTask(Action)` — запуск фоновой задачи с автоматическим перехватом и выводом ошибок в консоль.
4. `SafeTask(Func<Task>)` — асинхронный запуск фоновой задачи.
5. `SafeTaskWithUI<T>(Func<T>, Action<T>)` — фоновый расчёт с последующим выполнением в UI-потоке.
6. `RunUI(Action)` — выполнение кода в основном (UI) потоке Renga.

---

### Вариант 1. Простой скрипт (Top-Level Script)
Идеально подходит для простых и быстрых действий. Код пишется прямо «в теле» файла без классов и пространств имен.

**Пример:**
```csharp
using System.Windows.Forms;

if (RengaApp.Project != null)
{
    Print("Работает!");
    MessageBox.Show($"Открыт проект: {RengaApp.Project.ProjectInfo.Name}", "Информация");
}
```
* **Особенности:**
  * Переменные и функции `RengaApp`, `Print`, `SafeTask`, `SafeTaskWithUI`, `RunUI` уже автоматически доступны в глобальном контексте (через объект `ScriptGlobals`).
  * Неявные `using System;`, `System.IO`, `System.Collections.Generic`, `System.Linq`, `System.Threading.Tasks` подключены автоматически — их можно не писать.
  * Если в папку положить несколько `.cs` файлов, они будут просто склеены плагином в один длинный файл по алфавиту.
  * **Нюанс:** При склеивании файлов возможен конфликт имен переменных. Например, если в двух файлах объявить переменную `var button = ...` на глобальном уровне, скрипт выдаст ошибку компиляции.

---

### Вариант 2. Скрипт на классах (Class-Based Script)
Идеально подходит для сложных, больших и многофайловых скриптов. Плагин компилирует его как полноценную сборку (Assembly).

**Пример:**
```csharp
using System;
using System.Windows.Forms;
using Renga;

namespace MyScripts 
{
    public class MyCommand 
    {
        // Точка входа для плагина
        public void Execute(IApplication RengaApp, Action<string> Print) 
        {
            if (RengaApp.Project != null)
            {
                Print("Успешно выполнено через компиляцию C# класса!");
                Print("Проект: " + RengaApp.Project.ProjectInfo.Name);
            }
        }
    }
}
```
* **Особенности:**
  * **Точка входа:** Плагин ищет любой класс, содержащий метод `Execute(IApplication, Action<object>)` или `Execute(IApplication, Action<string>)`.
  * **Изоляция:** Переменные и методы инкапсулированы внутри классов. Вы можете создавать сколько угодно файлов с одинаковыми именами внутренних переменных — конфликтов не будет.
  * **IDE Поддержка:** Отличная работа автодополнения (IntelliSense) в Visual Studio / Rider без дополнительных настроек.
  * **Важный нюанс:** В этом стиле **нельзя писать плоский код** (top-level statements) вне классов. Весь исполняемый код должен находиться внутри методов. Если плагин найдет хотя бы одну глобальную инструкцию (например, вызов `Print(...)` вне методов), он автоматически переключится на *Вариант 1 (простой скрипт)* для сохранения совместимости.

---

### Использование папки `lib` (ВАЖНО!)

В папку `lib` вы можете складывать общие `.dll` библиотеки или файлы с C# кодом. 
Но **по умолчанию плагин не загружает `lib` в скрипты**, чтобы компиляция была мгновенной!

Если вашему скрипту нужны библиотеки из папки `lib`, добавьте в самое начало вашего `.cs` файла специальный комментарий:

```csharp
// @UseLib
using Renga;
using MyCustomLibrary; // библиотека из папки lib

Print("Этот скрипт использует общие библиотеки!");
```

Плагин увидит `// @UseLib` и автоматически подключит все `.dll` и код из папки `lib` при компиляции (как для простых скриптов, так и для скриптов на классах).

---

### Запуск готовых .dll как самостоятельных программ

Если ваша `.dll` — это уже полностью готовая программа (например, вы написали всю логику работы с Renga внутри нее), и вы хотите использовать кнопку `ScriptManager` просто как **ярлык для её запуска**, вам понадобится крошечный файл `script.cs`, который сыграет роль "пускового крючка". Сама по себе кнопка без текстового скрипта не отработает.

**Способ 1: Запуск через `//@UseLib` (Рекомендуемый)**
Положите вашу скомпилированную `MyPlugin.dll` в папку `Scripts/lib/`. Внутри этой DLL у вас должен быть класс и метод, с которого всё начинается. Создайте в папке кнопки файл `script.cs` с таким кодом:

```csharp
//@UseLib
using MyPlugin; // Ваше пространство имен из DLL

// Вызываем стартовый метод вашей DLL, передавая RengaApp (если нужно)
MainApp.Start(RengaApp);
```

**Способ 2: Запуск через Рефлексию (Без папки `lib`)**
Если вы хотите, чтобы `.dll` лежала прямо в папке с кнопкой (не используя `lib`), можно подгрузить её динамически прямо из скрипта кнопки:

```csharp
using System;
using System.IO;
using System.Reflection;

// Получаем путь к папке кнопки (где лежит этот скрипт и DLL)
string scriptFolder = Path.GetDirectoryName(ScriptManagerPlugin.ScriptHelper.GetCurrentScriptPath());
string dllPath = Path.Combine(scriptFolder, "MyPlugin.dll");

// Подгружаем вашу DLL в память
Assembly myAppAsm = Assembly.LoadFrom(dllPath);

// Ищем класс и стартовый метод (укажите полное имя класса с пространством имен)
Type mainAppType = myAppAsm.GetType("MyPlugin.MainApp");
MethodInfo startMethod = mainAppType.GetMethod("Start", BindingFlags.Public | BindingFlags.Static);

// Запускаем метод, передавая RengaApp
startMethod.Invoke(null, new object[] { RengaApp });
```

### Безопасная работа с фоновыми потоками и задачами (`SafeTask`)

Если скрипту требуется выполнить тяжелую операцию в фоновом потоке, обычный вызов `Task.Run` при возникновении ошибки может привести к аварийному завершению работы Renga. Для безопасного выполнения фоновых задач используйте `SafeTask`:

* В **Варианте 1 (простой скрипт)** функция доступна напрямую:
  ```csharp
  SafeTask(() => {
      // фоновая работа
      throw new Exception("ошибка потока"); // будет безопасно поймана и выведена в консоль
  });
  ```
* В **Варианте 2 (скрипт на классах)** используйте статический класс:
  ```csharp
  ScriptManagerPlugin.ScriptHelper.SafeTask(() => {
      // фоновая работа
  });
  ```

> [!WARNING]
> **Важное ограничение Renga API (COM Threading):**
> Renga API является однопоточным (STA). Вызывать любые свойства или методы `RengaApp` (и других объектов Renga) напрямую внутри фоновой задачи `SafeTask` **запрещено** — это приведет к ошибке `COMException` или мгновенному вылету/зависанию Renga.
> 
> Для фонового расчета с последующей записью данных в Renga используйте метод `RunUI` для возврата в поток интерфейса:
```csharp
SafeTask(() => {
    // Тяжелые вычисления (например, парсинг Excel или JSON) в фоне
    var result = CalculateSomething();
    
    // Возврат в UI-поток для работы с Renga API
    RunUI(() => {
        RengaApp.Project.ProjectInfo.Name = result;
    });
});
```

Либо используйте специализированный метод `SafeTaskWithUI`, который принимает две лямбды: фоновую (вычисляющую значение) и UI (принимающую результат и выполняющуюся в основном потоке Renga):
```csharp
SafeTaskWithUI(
    bgAction: () => {
        // Тяжелые вычисления в фоне (возвращаем результат)
        return CalculateSomething();
    },
    uiAction: (result) => {
        // Выполняется автоматически в UI-потоке Renga
        RengaApp.Project.ProjectInfo.Name = result;
    }
);
```

### Управление памятью и выгрузка сборок (`AssemblyLoadContext`)

**Все скрипты** (как на классах, так и простые Top-Level) компилируются в изолированном контексте `CollectibleAssemblyLoadContext` (доступно в .NET 8). При каждом изменении исходного кода скрипта плагин автоматически:
1. Выгружает предыдущую версию сборки из памяти (`Unload()`).
2. Компилирует и загружает обновленный вариант сборки (сам процесс компиляции происходит в фоновом потоке, чтобы не замораживать интерфейс Renga).

*Примечание:* Очистка выгруженных типов из памяти (Garbage Collection) происходит естественным путем силами .NET в фоне. Мы намеренно отказались от агрессивного вызова `GC.Collect()` при каждой перекомпиляции, так как это приводило к фризам хост-приложения (Renga).
